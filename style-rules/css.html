Оформление кода CSS

Правила оформления

  • Правильное именование классов.

    HTML5 уже избавляет нас от необходимости использования большого количества классов, т. к. можно (и нужно) обойтись соответствующим тэгом без класса, а также в лице именования этих тэгов даёт пример правильного именования классов. Классы должны называться по назначению элемента, а не по его внешнему виду, применённым стилям оформления или расположению в документе.

    Так, неправильным будет называть класс «red», «first», «left» и тому подобное. У любого элемента есть назначение, в соответствии с содержимым — будь то список навыков «skills», или раздел с информацией о компании «about‐us». Или в соответствии с акцентуацией контента, хотя это скорее частный случай — например, параграф, который должен быть выделен особым образом относительно других, может получить класс «emphasis».

    Также, есть разновидность классов, которые описывают состояние элемента. Примерами таких классов являются встроенные в браузер псевдо‐классы (:hover, :active, :focus, :visited) — простое слово, описывающее состояние некоего элемента. Аналогично им, может потребоваться создать собственные подобные классы, например: .visible (для элемента, который может быть скрыт, или отображаться), .disabled (для элемента, который может становиться неактивным), .active (для элемента, который может включаться, выбираться, или как‐то ещё становиться «активным»), .current (для элемента, который выбран в данный момент из некоторого набора), и другие.

    Также не стоит забывать о том, что имена классов оформляются строчными буквами и разделяются дефисом: «some‐class‐name».

  • Недопущение паразитных наследований.

    Оформляя CSS к разметке, стоит учитывать, что одни и те же тэги могут использоваться в разных местах (например, тэги arcticle, section, header, footer) и даже с теми же классами (например, ul.menu, a.email). По этой причине, прописывая стили оформления для элементов, нужно чётко указывать на какой именно элемент или группу элементов они будут распространяться.

    Таким образом нужно указывать прямую вложенность для элемента вёрстки от самого тэга html. Почему от html? Потому что классы у html так же используется в вёрстке CSS, как и другие тэги, но только эти классы не прописываются вручную, а используются различными скриптами, определяющими параметры браузера. Таким образом, html используется для представления состояния браузера (операционная система пользователя, браузер и его версия, поддерживаемые браузером возможности), body используется для представления состояния текущей страницы (например, специальный вид элементов, цветовая гамма и т. д. для определённой страницы), а остальные тэги уже более универсальные и обрабатывают частные состояния. По этой же причине (универсальности последующих тэгов), самый поздний тэг, с которого можно начать цепочку — это body.

    Пример — мы задаём оформление для тэга header на странице. Если мы укажем header {height: 300px}, то все элементы header, где бы они не находились, так же получат высоту в 300px, что очевидно неправильно. Но если мы указываем html > body > header {height: 300px}, то это уже повлияет только на header внутри body. Стоит отдельно заменить ошибочность записи html body header {height: 300px}, так как здесь не используется наследственный комбинатор и высота 300px, так же, как и в первом случае, применится ко всем элементам header на странице, разница лишь в том, что они должны находиться где‐нибудь внутри body, что и без того является необходимым условием нормальной вёрстки.

    Кроме того, это обеспечивает расширяемость вёрстки, возможность коллективной работы над вёрсткой, простоту добавления новых элементов, переносимость кода. Так, вам не нужно бояться, что добавляя стили оформления для нового элемента, это может как‐то затронуть уже существующие (или написанные кем‐то другим) элементы, или мучиться с необходимостью переназначить все ненужные вам стили, доставшиеся «в наследство» от других элементов (а ведь с внесением изменений в любой из этих элементов, таких стилей может прибавиться и вся вёрстка может непредсказуемым образом исказиться).

  • Однозначное определение области видимости стиля оформления.

    В дополнение к предыдущему пункту — нам всё‐же не всегда нужно ограничивать применение стилей оформления только одним конкретным элементом, поэтому составляя селектор, нужно руководствоваться требуемой областью видимости. Так, мы можем настроить параметры параграфа, заголовков или таблиц для всего документа сразу, для чего нам не нужно указывать вложенность, достаточно будет p {…} h1 {…} h2 {…} table {…}. Или нам нужно задать особый стиль для всех элементов на любом уровне внутри какой‐нибудь конструкции, например ссылки внутри подвала страницы: html > body > footer a {…} — мы пропускаем наследственный комбинатор перед a, и таким образом это коснётся всех ссылок внутри html > body > footer.

    Но во всех остальных случаех, то есть в их подавляющем большинстве, нужно чётко определять целевой элемент путём указания полного наследования.

  • Учёт каскадности.

    Следует учитывать и правильно использовать свойство каскадности. Так, параметры текста каскадно наследуются и нет необходимости их заново задавать, если мы их уже задали для вышестоящего элемента. Для свойств, значение которых не наследуется, можно форсировать получение значения, аналогичного родительскому, указав ключевое слово inherit в качестве значения.

  • Правильное оформление кода.

    Код должен удовлетворять следующим соглашениям по оформлению:

    • Отступ (inherit) должен выполняться символом табуляции (редактор должен быть настроен правильно, чтобы не вставлять пробелы вместо табуляции).
    • Каждый отдельный селектор записывается с новой строки:
      html > body > header,
      html > body > footer
      {
          width: 980px;
      }
    • Фигурные скобки ставятся друг под другом на новой строке, содержимое внутри определённого ими блока пишется с отступом (см. пример выше).
    • После двоеточия в названии свойства ставится пробел, после значения — точка с запятой:
      width: 980px;
    • В свойствах с набором значений, каждое значение в наборе пишется с новой строки с отступом:
      {
          transition: opacity 0.3s,
              transform 0.3s;
      }
    • Каждое свойство пишется на новой строке, отделять один набор свойств от другого следует пустой строкой.

  • Правильный порядок свойств и соответствие другим правилам из  css-props.html.

    Так как к элементу может быть применено большое количество свойств, чтобы не путаться, избежать возможного дублирования, быстро находить нужное свойство и при работе с одним свойством, иметь перед глазами близкие к нему по назначению и могущие оказывать влияние другие свойства, следует придерживаться данных правил группировки.
